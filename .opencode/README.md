# OpenCode Adapter

This folder has two roles depending on where you are reading it:

- In this package repo, `.opencode/` contains the tracked OpenCode plugin and agent/skill reference surface.
- In a consumer project after `hf-setup --command install`, `hf-setup --command init`, or `hf-setup --command sync`, `.opencode/` is generated managed output derived from the installed package.

OpenCode-specific material belongs here; the framework itself lives at the repo root:

- `agents/`
- `subagents/`
- `skills/` for the active reusable procedures only
- `src/`
- `schemas/`

See `../README.md` for the canonical framework surface and `../plans/README.md` for the active plan-doc format.

`plugins/hybrid-runtime.js` is the managed plugin loader. It is generated by the lifecycle commands and should not be edited by hand in consumer projects.

`registry.json` is the managed asset registry that maps canonical agents, subagents, and skills into the OpenCode adapter surface.

## Consumer Project Behavior

For the general consumer install and lifecycle contract, see [`docs/consumer-install.md`](../docs/consumer-install.md).

When a consumer project runs the OpenCode install or init lifecycle, the package manages `.opencode/` in the target project by:

- generating `.opencode/plugins/hybrid-runtime.js` (the plugin loader)
- generating `.opencode/registry.json` (the asset registry)
- optionally generating `.opencode/agents/` and `.opencode/skills/` when `assets.opencode.syncGenerated` is enabled

Re-running the OpenCode sync lifecycle refreshes the managed OpenCode surface from the installed package without re-scaffolding `plans/` or `vault/`.

### OpenCode lifecycle commands

Use `hf-setup` with `--platform opencode` when a consumer project only needs OpenCode and should not install Claude at all:

| Purpose | `hf-setup` command | Combined equivalent |
|---|---|---|
| Install OpenCode wiring into an existing project | `hf-setup --command install --platform opencode` | `hf-setup --command install` |
| Scaffold `plans/` + `vault/`, then install OpenCode wiring | `hf-setup --command init --platform opencode` | `hf-setup --command init` |
| Refresh managed OpenCode output | `hf-setup --command sync --platform opencode` | `hf-setup --command sync` |
| Remove managed OpenCode output | `hf-setup --command uninstall --platform opencode` | `hf-setup --command uninstall` |

The combined commands remain valid when the project wants both adapters. The `--platform opencode` flag exists so an OpenCode-only consumer never needs to think about Claude.

If a consumer project sets `assets.opencode.copy` in `hybrid-framework.json`, `scripts/sync-opencode-assets.mjs` can also materialize selected canonical markdown assets into adapter-local reference folders:

- `agents/` and `subagents/` entries are written to `.opencode/agents/`
- `skills/` entries mirror their canonical path under `.opencode/skills/`
- `assets.mode` uses the same `copy` / `symlink` / `references` rules as Claude, with `references` falling back to copies when symlinks are unavailable

Those OpenCode markdown mirrors are generated references for the consumer project; the root package assets remain canonical.

In consumer projects, treat `.opencode/` as generated output:

- customize behavior through `hybrid-framework.json`
- run `hf-setup --command sync --platform opencode` (or `hf-setup --command sync`) after changing adapter asset settings or upgrading the package
- run `hf-setup --command uninstall --platform opencode` (or `hf-setup --command uninstall`) to remove managed OpenCode wiring cleanly while preserving unrelated local settings

## hf_search Tool

The OpenCode runtime plugin also registers the `hf_search` tool, which lets agents issue targeted semantic queries against the unified vault and code index:

```
hf_search(query: string, top_k?: number, source?: "vault" | "code" | "all")
```

See `docs/vault.md` for the full `hf_search` reference including the `source` filter, return format, and relationship to auto-injection.
