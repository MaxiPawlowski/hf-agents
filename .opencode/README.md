# OpenCode Adapter

This folder has two roles depending on where you are reading it:

- In this package repo, `.opencode/` is the adapter-specific surface that points back to canonical root assets.
- In a consumer project after `hf-install`, `hf-init`, or `hf-sync`, `.opencode/` is generated managed output derived from the installed package.

This folder is adapter-local only; it is not the source of truth for the framework.

Canonical assets live in the repo root:

- `agents/`
- `subagents/`
- `skills/`
- `src/`
- `schemas/`

See `../README.md` for the canonical framework surface and `../plans/README.md` for the active plan-doc format.

`registry.json` exists only to map OpenCode-facing asset IDs back to those active root files.

`.opencode/agents/` and `.opencode/skills/` are generated by
`scripts/sync-opencode-assets.mjs` and are not tracked in git in this repository.

## Consumer Project Behavior

For the general consumer install and lifecycle contract, see [`docs/consumer-install.md`](../docs/consumer-install.md).

When a consumer project runs the OpenCode install or init lifecycle, the package generates `.opencode/` in the target project with:

- `.opencode/plugins/hybrid-runtime.js` as the managed loader that re-exports the compiled runtime plugin
- `.opencode/registry.json` as the managed asset registry that maps OpenCode-facing entries back to canonical framework assets
- `.opencode/agents/` and `.opencode/skills/` when OpenCode asset sync is enabled

Re-running the OpenCode sync lifecycle refreshes those generated files from the installed package without touching the consumer project's `plans/` or `vault/` docs.

### OpenCode-only lifecycle commands

Use the dedicated OpenCode commands when a consumer project only needs OpenCode and should not install Claude at all:

| Purpose | Dedicated OpenCode command | Combined equivalent |
|---|---|---|
| Install OpenCode wiring into an existing project | `hf-install-opencode` | `hf-install --tool opencode` |
| Scaffold `plans/` + `vault/`, then install OpenCode wiring | `hf-init-opencode` | `hf-init --tool opencode` |
| Refresh managed OpenCode output | `hf-sync-opencode` | `hf-sync --tool opencode` |
| Remove managed OpenCode output | `hf-uninstall-opencode` | `hf-uninstall --tool opencode` |

The combined commands remain valid when the project wants both adapters. The OpenCode-only aliases exist so an OpenCode-only consumer never needs to think about Claude.

Sync rules are driven by the consumer project's `hybrid-framework.json`:

- `assets.opencode.syncGenerated` defaults to `true` and keeps the packaged OpenCode agent and skill surface materialized under `.opencode/agents/` and `.opencode/skills/`.
- `assets.opencode.copy` can add extra canonical markdown assets from `agents/`, `subagents/`, or `skills/` into the generated OpenCode surface without editing package internals.
- `assets.mode` chooses how those generated files are written: `copy` always copies, `symlink` requires symlinks, and `references` prefers symlinks but falls back to copying when the local environment does not allow links.
- Agent and subagent markdown land in `.opencode/agents/`; skill markdown mirrors its canonical path under `.opencode/skills/`.

`.opencode/plugins/hybrid-runtime.js` re-exports the compiled OpenCode plugin from `dist/src/opencode/plugin.js`, is generated by install, and is not tracked in git.

In consumer projects, treat `.opencode/` as generated output:

- customize behavior through `hybrid-framework.json`
- run `hf-sync-opencode` (or `hf-sync --tool opencode`) after changing asset config or upgrading the package
- use `hf-uninstall-opencode` (or `hf-uninstall --tool opencode`) to remove managed OpenCode artifacts cleanly

`scripts/install-runtime.mjs --tool opencode` rewrites that loader and runs `scripts/sync-opencode-assets.mjs` so the adapter surface stays aligned with the root prompt assets.
